# Cron & Rebuild Failure Ledger

Operational incident log for brain-platform GHA workflows and the daily brain rebuild. One row per incident. **Newest-first.** Update `Status` in place when resolved; add new rows at the top.

**Auto-capture is enabled.** New rows with `Status: OPEN` and an `_auto-captured; pending triage_` Root Cause are written by `.github/workflows/log-cron-incident.yml` when a watched workflow fails. The same workflow flips `OPEN` → `RESOLVED (auto)` when the next scheduled run of the same workflow succeeds. Auto-resolve never touches `BLOCKED`, `FLAKE`, or human-set `RESOLVED` rows. To disable, set repo variable `CRON_INCIDENT_LOGGER_ENABLED=false` (full kill) or `CRON_INCIDENT_AUTO_RESOLVE_ENABLED=false` (keep capture, drop auto-resolve). The sentinel comments below are the contract — do not delete or reorder them.

## Recurring Patterns

Known failure classes that have hit more than once. Check here before assuming a new incident is novel.

### Secret wired in repo but not passed to workflow

**Pattern:** A secret exists in `Settings → Secrets and variables → Actions` but is not listed in the workflow's `env:` block. The pipeline runs, reads `os.environ[...]`, and either silently falls back or throws at startup.

**Seen in:** `FRED_API_KEY` (daily-rebuild, 2026-05-26/27), `SUPABASE_S3_ENDPOINT` (6 Tier-1 workflows, 2026-05-26), `FIRECRAWL_API_KEY` (collier-permits, 2026-05-27).

**Check:** when adding a new secret, grep all workflows that invoke the same pipeline for the secret name. Adding to `gh secret set` is step 1; adding to every relevant `env:` block is step 2.

---

### Corridor rename without alias sync

**Pattern:** A corridor slug is renamed in `corridor_profiles` (and propagated to `fixtures/corridor-rents.json` + `fixtures/corridor-centroids.json`) but `refinery/lib/corridor-aliases.mts` is not updated. CI goes red immediately on the next push — the alias coverage test asserts every cre-swfl slug has an explicit entry.

**Seen in:** Bonita Beach Rd + Daniels Pkwy renames (Step 4.5, 2026-05-27 commit `d3db10c`).

**Check:** any time a corridor is renamed in the DB, run `bun test refinery/lib/corridor-aliases.test.mts` locally before pushing. The fix is always one-line per renamed corridor in `CORRIDOR_ALIASES`.

---

### Leaf / conditional metric slug emitted but never registered in the vocab

**Pattern:** A leaf pack emits a `key_metrics` slug that is not in `refinery/vocab/brain-vocabulary.json`. The leaf builds fine (0 orphans at its own stage 2.5). But when that brain feeds master via the thin pipe, master's _strict_ stage-2.5 normalize hits the unregistered slug → `Orphan Concept error` → deterministic HOLD (exit 1), prior `master.md` keeps serving. Surfaces only when master re-synthesizes (i.e. when the leaf goes stale and rebuilds), so it can lag the offending commit by days. The _conditional_ variant is worse: a slug emitted behind an `if` (only when its data exists) is invisible to every check until the data lands, then orphans master with zero warning.

**Seen in:** `econ-dev-swfl` — `econ_dev_announcements_90d` / `econ_dev_announcements_prior_90d` (shipped at `f8e3037`/`860881b` without vocab; held the 2026-06-07 08:58 UTC rebuild, run 27087990541). Root cause was masked because the pre-push gate ran the **master-only** coverage check, which never inspects leaf slugs. Earlier instance: `housing_months_of_supply_swfl` (2026-06-03, conditional-metric orphan).

**Check:** when you touch `refinery/packs/**` or `refinery/vocab/**`, register AND document **every** slug the pack can emit (concept with `prefLabel` + `scope_note` + a `slug_index` identity entry), including conditional ones, in the same commit; then run `bun refinery/tools/check-vocab-coverage.mts --all` (the `--all` is mandatory — the bare default only checks master). The pre-push hook now enforces both `--all` and a source scan of touched packs for unregistered double-quoted `metric:` literals (the conditional-slug guard). Full rule: `CLAUDE.md` RULE 1 breaker #2.

### bun.lock drift after package.json edit

**Pattern:** A package is added or removed from `package.json` without re-running `bun install` and committing the updated `bun.lock`. CI hits `bun install --frozen-lockfile`, detects the drift, and exits in under 1s with `error: lockfile had changes, but lockfile is frozen`. The auto-resolve listener only fires on `event == 'schedule'`, so manual-dispatch failures pile up as OPEN rows until tonight's cron.

**Seen in:** `@sanity/client` removal (`e6258d0`, 2026-06-01, 3 manual dispatches failed). Fixed by `bd06ff0`.

### Flaky test reddens `main` at random (non-deterministic assertion)

**Pattern:** A test with a non-deterministic input — a crypto digest, `Date.now()`, `Math.random()`, `crypto.randomUUID()` — asserts something that only holds for _some_ of the random values. It passes locally most of the time and reddens CI at random, **independent of the diff that triggered the run.** No pre-push gate can stop it: it would pass at push time (most of the time) and fail in CI later. The tell: `main` goes red on a test that has nothing to do with the commit that triggered the run. The fix is ALWAYS to make the test deterministic — never to gate harder or to retry.

**Seen in:** `proposal-nonce` "tampered signature is rejected" (2026-06-13, surfaced on `72465f0` / GATE B floor — an unrelated zhvi/zori diff). The test flipped the FINAL base64url char of a 32-byte HMAC signature; the last char of a 43-char base64url string carries only 4 meaningful bits (the low 2 are decode-ignored padding), so `A` (`000000`) ↔ `B` (`000001`) decode to the IDENTICAL 32 bytes — the "tampered" token still verified, `r.ok===true`, assertion failed. Measured **6.52% over 5000 runs** (≈ 1/16; the encoder emits only 16 possible final chars). Held red `main` across 6 commits / ~2h until fixed.

**Check:** when `main` is red on a test unrelated to the triggering commit, loop the test locally (`for i in $(seq 1 30); do bun test <file> >/dev/null 2>&1 || echo FAIL; done`) to measure the flake rate BEFORE assuming the diff broke it. Fix the non-determinism in the assertion (here: flip a decoded digest byte, not a base64url char). When writing crypto/time/random tests, grep the new test for `Date.now()|Math.random|randomUUID|new Date()` feeding an assertion.

### Pack ⇆ catalog (or per-pack assertion) drift with no pre-push gate

**Pattern:** A `refinery/packs/**` edit changes a pack's domain/scope/ttl or its `sources`, but the leaf `catalog.mts` (the MCP capability-inventory mirror) or the pack's own `bun:test` ("sources wired") is not updated in the same commit. `bun test` goes red in CI, but the pre-push gate (pre-2026-06-13) ran only lockfile + vocab + alias — never the pack mirror — so red `main` sat unnoticed.

**Seen in:** redfin-lee parity build (`d9aa670`, 2026-06-13) — `BRAIN_CATALOG: domain/scope/ttl_seconds match the pack definition` + `propertiesLeeValue pack: source connectors wired` both red; sat ~2h across 5 pushes.

**Check:** the pre-push hook now runs **Gate 5** — `catalog.test.mts` (env-safe mirror, hard-block) on ANY `refinery/packs/**` change, plus each touched pack's `bun:test` (additive). vitest view-parity tests (zhvi/zori GATE A + `*-view-equivalence`) spawn a DuckDB/Postgres subprocess that only resolves in CI, so they are skipped locally — never blocked. To self-check before a pack push: `bun test refinery/packs/catalog.test.mts`.

**Check:** after any `package.json` change, run `bun install` locally and `git add bun.lock` before pushing. If CI is red with this error, the fix is always the same: `bun install && git add bun.lock && git commit -m "fix(lockfile): regenerate bun.lock" && git push`.

**Stop worrying about it if:** the last scheduled cron run (check `gh run list --workflow=daily-rebuild.yml --limit=3`) shows `success` — the OPEN rows are just waiting on the auto-resolve timer, not an active breakage.
