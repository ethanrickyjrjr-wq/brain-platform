# Cron & Rebuild Failure Ledger

Operational incident log for brain-platform GHA workflows and the daily brain rebuild. One row per incident. **Newest-first.** Update `Status` in place when resolved; add new rows at the top.

**Auto-capture is enabled.** New rows with `Status: OPEN` and an `_auto-captured; pending triage_` Root Cause are written by `.github/workflows/log-cron-incident.yml` when a watched workflow fails. The same workflow flips `OPEN` → `RESOLVED (auto)` when the next scheduled run of the same workflow succeeds. Auto-resolve never touches `BLOCKED`, `FLAKE`, or human-set `RESOLVED` rows. To disable, set repo variable `CRON_INCIDENT_LOGGER_ENABLED=false` (full kill) or `CRON_INCIDENT_AUTO_RESOLVE_ENABLED=false` (keep capture, drop auto-resolve). The sentinel comments below are the contract — do not delete or reorder them.

Status key: `RESOLVED` | `RESOLVED (auto)` (auto-flipped by the listener; not human-confirmed) | `OPEN` | `BLOCKED` | `FLAKE` (transient, retry-able, no code change)

## Recurring Patterns

Known failure classes that have hit more than once. Check here before assuming a new incident is novel.

### Secret wired in repo but not passed to workflow

**Pattern:** A secret exists in `Settings → Secrets and variables → Actions` but is not listed in the workflow's `env:` block. The pipeline runs, reads `os.environ[...]`, and either silently falls back or throws at startup.

**Seen in:** `FRED_API_KEY` (daily-rebuild, 2026-05-26/27), `SUPABASE_S3_ENDPOINT` (6 Tier-1 workflows, 2026-05-26), `FIRECRAWL_API_KEY` (collier-permits, 2026-05-27).

**Check:** when adding a new secret, grep all workflows that invoke the same pipeline for the secret name. Adding to `gh secret set` is step 1; adding to every relevant `env:` block is step 2.

---

### Corridor rename without alias sync

**Pattern:** A corridor slug is renamed in `corridor_profiles` (and propagated to `fixtures/corridor-rents.json` + `fixtures/corridor-centroids.json`) but `refinery/lib/corridor-aliases.mts` is not updated. CI goes red immediately on the next push — the alias coverage test asserts every cre-swfl slug has an explicit entry.

**Seen in:** Bonita Beach Rd + Daniels Pkwy renames (Step 4.5, 2026-05-27 commit `d3db10c`).

**Check:** any time a corridor is renamed in the DB, run `bun test refinery/lib/corridor-aliases.test.mts` locally before pushing. The fix is always one-line per renamed corridor in `CORRIDOR_ALIASES`.

---

### bun.lock drift after package.json edit

**Pattern:** A package is added or removed from `package.json` without re-running `bun install` and committing the updated `bun.lock`. CI hits `bun install --frozen-lockfile`, detects the drift, and exits in under 1s with `error: lockfile had changes, but lockfile is frozen`. The auto-resolve listener only fires on `event == 'schedule'`, so manual-dispatch failures pile up as OPEN rows until tonight's cron.

**Seen in:** `@sanity/client` removal (`e6258d0`, 2026-06-01, 3 manual dispatches failed). Fixed by `bd06ff0`.

**Check:** after any `package.json` change, run `bun install` locally and `git add bun.lock` before pushing. If CI is red with this error, the fix is always the same: `bun install && git add bun.lock && git commit -m "fix(lockfile): regenerate bun.lock" && git push`.

**Stop worrying about it if:** the last scheduled cron run (check `gh run list --workflow=daily-rebuild.yml --limit=3`) shows `success` — the OPEN rows are just waiting on the auto-resolve timer, not an active breakage.
