Rule 0
Tier Decision — where data goes
Every dataset lands in exactly ONE tier.
  • Tier 1 — Cold (S3/Parquet): over 10k rows, archival, occasional queries, or bulk historical. Path: lake-tier1/{domain}/{table}.parquet. DuckDB queries at runtime.
  • Tier 2 — Hot (Postgres): under ~10k rows, consumed by a brain on every run, needs SQL joins. Schema: data_lake.* in Supabase.
Over 10k rows OR purely archival → Tier 1.
Small + consumed by a brain + needs joins → Tier 2.
No consuming brain yet → Tier 1 (never put data in Postgres with no reader).
Rule 1
DLT vs DuckDB — different jobs, not interchangeable
  • DLT: Python, extracts and LOADS data into the lake. Lives in ingest/pipelines/ (T2) or ingest/duckdb_pipelines/ (T1). Each pipeline: pipeline.py, resources.py, constants.py.
  • DuckDB: TypeScript, READS Parquet from S3 at query time. Does not load data. Lives in tools/lake-mcp-server.mts. READ ONLY on Postgres attachment.
Flow: Source → DLT → T1 (Parquet) → DuckDB reads it
                → T2 (Postgres) → Brains read via SQL
Rule 2 — CRITICAL
Firecrawl/Spider Wrapper — USE IT for all HTML scraping
Every HTML-scraping pipeline must go through ingest/lib/extract_client.py — never call vendor clients directly.
  • scrape_with_fallback(url) — plain page-to-markdown. Firecrawl primary → Spider fallback. Use everywhere firecrawl_client.scrape() was used before.
  • extract(prompt, urls=[...], schema={}) — AI-structured extraction. Firecrawl /v2/agent primary → Spider /ai/scrape per-URL fallback. Use everywhere firecrawl_client.agent() was used before.
  • scrape_with_actions() — Accela click-through ONLY. Call firecrawl directly. No Spider analogue.
GHA workflows using the wrapper need BOTH secrets:
FIRECRAWL_API_KEY: ${{ secrets.FIRECRAWL_API_KEY }}
SPIDER_API_KEY: ${{ secrets.SPIDER_API_KEY }}
If SPIDER_API_KEY is missing, wrapper runs firecrawl-only with a warning (does not crash).
Never call firecrawl_client.scrape(), firecrawl_client.agent(), or spider_client.scrape() directly in pipeline code. They bypass the fallback layer and will silently return zero rows on failure.
Rule 3
Step-by-step: every new pipeline PR
  • Source recon: pull actual API docs, confirm URL, cadence, geography.
  • Decide tier (see Rule 0). No brain yet → Tier 1.
  • Scaffold ingest/pipelines/<name>/ (T2) or ingest/duckdb_pipelines/<name>/ (T1).
  • Add entry to ingest/cadence_registry.yaml under not_yet_running: first.
  • Create .github/workflows/ingest-<name>.yml — cron + workflow_dispatch + --dry-run. Stagger timing away from 13:30 UTC band.
  • Call upsert_inventory_row() from ingest/lib/tier1_inventory.py on success (T1).
  • Move registry entry from not_yet_running: to pipelines: after first successful cron run.
  • Wire freshness token in consuming brain + citation via refinery/lib/citation-url.mts.
  • Update SESSION_LOG.md and this inventory.
Rule 4
What to keep record of — per pipeline
These are the things that get lost between sessions and cause re-work.
  • cadence_registry.yaml — dlt_schema_name must be read from live data_lake._dlt_loads table, NOT assumed from pipeline_name=. They often differ.
  • Row count on first load — log in cadence_registry comment.
  • Date of first successful GHA cron run — log as last_known_good.
  • Column renames — DLT mangles reserved words. Document in constants.py. Known: FDOT YEAR → yearx.
  • Any UNIQUE constraint migration — log in docs/sql/ and SESSION_LOG.
  • Vendor-specific gotchas — e.g., FL DOR Form 3 had 1900-era date serials in early fiscal years; needed _fix_bad_dates.py.
Rule 5
Common failure patterns — read before debugging
  • Pagination cap: ArcGIS 1000-row limit. Use .range() loop + .order(), check exceededTransferLimit.
  • FIPS mismatch: 5-digit vs 3-digit county codes. Normalize at ingest.
  • Citation hardcoded: Never hardcode "data_lake." in citation strings. Branch on env.source. (Source: env-swfl PHANTOM DATA incident.)
  • Secret wired but not passed: GHA secret exists but workflow env: block doesn't reference it.
  • dlt_schema_name wrong: Always read from live _dlt_loads — never assume it matches pipeline_name=.
  • Scraper returns zero rows silently: You used the vendor client directly instead of the wrapper. Firecrawl can return status=completed with data=null without erroring.
  • Census API silent 204: time_slot_id as predicate causes 204. Use requests.get(url) not params= (encodes + as %2B).
